# HR Assistant

## 1. Контекст проекта

- Продукт: сервис для подготовки к собеседованиям с личным профилем пользователя, чатами-тренажёрами, генерацией вопросов и автоматической оценкой ответов.
- Целевая аудитория: соискатели в IT и digital-направлениях, которым нужно тренировать ответы перед интервью.
- Основная проблема: пользователю сложно регулярно практиковаться, отслеживать прогресс и понимать, становится ли он сильнее от сессии к сессии.
- Текущий этап: MVP с рабочим backend, авторизацией, профилем пользователя, чатами интервью и модулем геймификации.

## 2. Что было ДО изменений

- Пользователь мог зарегистрироваться и войти в систему, но не мог получить свой профиль и обновить его через удобные API-эндпоинты.
- Чаты существовали отдельно от пользователя. Это означало, что система не разделяла данные по владельцам, и нельзя было надёжно строить персональную историю работы.
- Интервью-чат умел:
  - создавать чат
  - получать вопрос
  - сохранять ответ
  - отправлять ответ во внешний сервис оценки
- Система не фиксировала пользовательскую активность по дням.
- Не было механики удержания:
  - streak по дням
  - ачивки
  - счётчики сообщений
  - счётчики созданных чатов
- Документация API была разрозненной и не объясняла назначение каждого эндпоинта.

## 3. Что МЫ ИЗМЕНИЛИ

### Изменение 1. Добавили эндпоинты профиля пользователя

- Что сделали:
  - реализовали `GET /api/users/profile`
  - реализовали `PATCH /api/users/profile`
- Почему сделали:
  - фронтенду нужен способ получать и редактировать данные текущего пользователя после авторизации
- Как работает сейчас:
  - профиль доступен только авторизованному пользователю по JWT
  - можно обновлять `firstName`, `lastName`, `username`, `position`, `techStack`, `password`
  - новый пароль при обновлении хешируется
- Какое влияние это дало:
  - профиль стал полноценной частью продукта, а не набором полей только на этапе регистрации
  - фронтенд получил стабильную точку для экрана профиля и редактирования данных

### Изменение 2. Исправили авторизацию для старых пользователей

- Что сделали:
  - добавили поддержку legacy-паролей, которые в старой базе были сохранены в открытом виде
- Почему сделали:
  - часть существующих пользователей не могла войти, потому что новая логика ожидала только bcrypt-хеши
- Как работает сейчас:
  - если пароль в базе хранится как bcrypt-хеш, выполняется `bcrypt.compare`
  - если пароль в базе хранится как обычная строка, выполняется прямое сравнение
- Какое влияние это дало:
  - старые тестовые и уже созданные аккаунты снова стали рабочими без миграции данных вручную
  - снизился риск блокировки демо-сценариев из-за несовместимости старой базы

### Изменение 3. Привязали чаты к пользователю

- Что сделали:
  - добавили связь `chat -> user`
  - перевели chat-эндпоинты в режим работы только с авторизованным пользователем
- Почему сделали:
  - без владельца чата нельзя построить персональный список интервью, историю и геймификацию
- Как работает сейчас:
  - создание чата сразу сохраняет владельца
  - получение списка чатов возвращает только чаты текущего пользователя
  - история, получение вопроса и отправка ответа теперь работают только для чата владельца
- Какое влияние это дало:
  - данные пользователей изолированы друг от друга
  - появилась основа для персональной статистики и достижений

### Изменение 4. Добавили совместимость со старыми чатами

- Что сделали:
  - реализовали автоматическую привязку legacy-чатов без `userId` к пользователю при первом обращении
- Почему сделали:
  - после ввода пользовательских чатов старые записи перестали отображаться
- Как работает сейчас:
  - если у пользователя ещё нет своих чатов, система ищет старые записи без владельца и привязывает их
  - если открывается конкретный старый чат без `userId`, он тоже автоматически закрепляется за текущим пользователем
- Какое влияние это дало:
  - старые данные не потерялись после изменения модели
  - не пришлось вручную чистить базу или писать отдельный одноразовый скрипт миграции

### Изменение 5. Добавили модуль геймификации

- Что сделали:
  - создали отдельный `GamificationModule`
  - добавили сущности:
    - `achievements`
    - `user_achievements`
    - `user_daily_activity`
    - `user_gamification_stats`
- Почему сделали:
  - продукту не хватало механики удержания и понятного прогресса пользователя
- Как работает сейчас:
  - при первом ответе за день фиксируется активный день
  - считается текущий streak
  - считается максимальный streak
  - считаются всего активные дни
  - считаются всего отправленные ответы
  - считаются всего созданные чаты
- Какое влияние это дало:
  - продукт получил систему регулярной мотивации, а не только разовые интервью-сессии
  - появилась база для экранов прогресса и персональных достижений

### Изменение 6. Внедрили ачивки

- Что сделали:
  - добавили набор достижений:
    - `First Step` за первый ответ
    - `On Fire 3` за 3 дня подряд
    - `On Fire 5` за 5 дней подряд
    - `On Fire 15` за 15 дней подряд
    - `Talker I` за 20 ответов за день
    - `Talker II` за 50 ответов за день
    - `Interviewer` за 10 созданных чатов
    - `Interview Architect` за 25 созданных чатов
    - `Marathon Speaker` за 100 ответов всего
- Почему сделали:
  - пользователю нужно не только проходить сессии, но и видеть конкретные цели и промежуточные результаты
- Как работает сейчас:
  - после отправки ответа или создания чата система пересчитывает статистику
  - если условие достижения выполнено, ачивка записывается в `user_achievements`
  - новые ачивки возвращаются в ответе `POST /api/chat/:id/answer`
- Какое влияние это дало:
  - увеличилась вовлечённость в повторные сессии
  - появился понятный триггер для возвращения в продукт на следующий день

### Изменение 7. Добавили API для геймификации

- Что сделали:
  - реализовали `GET /api/gamification/profile`
  - реализовали `GET /api/gamification/achievements`
- Почему сделали:
  - фронтенду нужен не просто backend-расчёт, а готовые данные для экранов прогресса и списка достижений
- Как работает сейчас:
  - `profile` возвращает статистику пользователя, активность за сегодня и уже полученные ачивки
  - `achievements` возвращает полный список всех достижений и статус открытия каждого из них
- Какое влияние это дало:
  - фронтенд может показывать отдельный экран прогресса
  - пользователь видит не только полученные ачивки, но и то, что ещё можно открыть

### Изменение 8. Оформили полную API-документацию

- Что сделали:
  - создали отдельный файл `APIDocs.md`
  - описали все рабочие эндпоинты и добавили пояснение, что делает каждый из них
- Почему сделали:
  - без документации фронтенду сложнее интегрироваться, а демонстрация проекта выглядит незавершённой
- Как работает сейчас:
  - в одном файле собраны auth, users, positions, technologies, chat и gamification эндпоинты
  - для каждого эндпоинта указано:
    - что делает
    - что принимает
    - что возвращает
- Какое влияние это дало:
  - ускорилась интеграция frontend/backend
  - стало проще использовать проект в демо, защите и дальнейшей разработке

## 4. Техническая часть

- Backend: NestJS
- Язык: TypeScript
- База данных: SQLite
- ORM: TypeORM
- Авторизация: JWT + Passport
- Валидация входных данных: `class-validator` + `ValidationPipe`
- Хеширование паролей: `bcrypt`
- Внешняя интеграция:
  - backend отправляет ответ пользователя во внешний сервис оценки по адресу `http://localhost:8000/evaluate`

Архитектурные решения:

- Модульная структура:
  - `auth`
  - `users`
  - `chat`
  - `positions`
  - `technologies`
  - `gamification`
- Персональные чаты:
  - каждый новый чат создаётся с привязкой к пользователю
- Отдельный слой геймификации:
  - статистика и достижения вынесены из chat-логики в отдельный сервис
- Совместимость со старыми данными:
  - legacy-чаты без владельца автоматически закрепляются за пользователем

Ограничения и компромиссы:

- Используется `synchronize: true` в TypeORM
  - это удобно для MVP, но рискованно для production
- SQLite подходит для локальной разработки и демо
  - но ограничивает масштабирование и конкурентную запись
- Оценка ответов зависит от внешнего локального сервиса на `:8000`
  - если он недоступен, интервью-оценка не сработает
- Активный день сейчас считается по первому отправленному ответу за день
  - вход без ответа не увеличивает streak

## 5. Результат

Фактические изменения в продукте:

- появился полноценный профиль пользователя с API получения и обновления
- чаты стали персональными, а не общими для всей базы
- история старых чатов сохранилась после изменения модели
- появилась игровая статистика:
  - streak
  - активные дни
  - число ответов
  - число чатов
- появились достижения, которые открываются автоматически
- фронтенд получил отдельные эндпоинты для прогресса и ачивок
- проект получил централизованную API-документацию

Почему это улучшило продукт:

- пользователь теперь видит не только один ответ на вопрос, а собственный прогресс во времени
- повторные заходы в систему стали мотивироваться не абстрактно, а через конкретные цели
- данные стали пользовательскими, поэтому продукт можно безопасно развивать как персональный сервис подготовки
- backend стал удобнее для интеграции с фронтендом, потому что ключевые сценарии закрыты API

## 6. Нереализованные идеи / проблемы

- Нет отдельного endpoint для сброса флага `isNew` у новых ачивок после просмотра.
- Нет экранной модели “уровней пользователя”, только счётчики и достижения.
- Нет аналитики качества ответов по категориям:
  - структура
  - уверенность
  - полнота
- Нет связи между профессией пользователя и типом задаваемых вопросов.
- Нет рекомендаций следующего вопроса на основе слабых мест.
- Нет миграций базы данных.
  - сейчас схема меняется через `synchronize`, что неудобно для контролируемых релизов
- Нет production-ready обработки ошибок внешнего сервиса оценки.
- Нет ролей администратора и пользовательских прав доступа.

- добавить endpoint для отметки ачивок как просмотренных
- вернуть в `POST /api/chat/:id/answer` полную информацию о новой ачивке, а не только `achievementId`

- улучшить ответы API по ошибкам для фронтенда
- убрать выдачу лишних полей в публичных эндпоинтах

### Среднесрочные (3–6 месяцев)

- добавить аналитику по качеству ответов пользователя

- добавить уровни прогресса и цели на неделю
- реализовать подбор сценариев интервью по профессии
- перейти с SQLite на PostgreSQL
- внедрить миграции вместо `synchronize`

### Долгосрочные (6+ месяцев)

- масштабировать продукт как персонального AI-помощника по подготовке к собеседованиям
- добавить адаптивные интервью-сценарии на основе прошлых ответов пользователя
- внедрить сравнение прогресса по неделям и месяцам
- добавить командный или образовательный сценарий использования для вузов и карьерных центров
- вывести продукт из режима локального MVP в облачную архитектуру с production-хранилищем и мониторингом

## 8. Видение продукта

- HR Assistant должен превратиться из простого чат-интерфейса с вопросами в персональную систему подготовки к собеседованиям.
- Главное отличие от базовых тренажёров вопросов должно быть в том, что продукт не просто задаёт вопрос, а:
  - запоминает историю пользователя
  - измеряет регулярность практики
  - показывает прогресс
  - мотивирует возвращаться
  - помогает готовиться именно под конкретную роль
- Ценность для пользователя:
  - регулярная практика
  - понимание собственного роста
  - снижение тревожности перед собеседованием
  - более структурированные и уверенные ответы
- Ценность для рынка:
  - продукт можно использовать как карьерный тренажёр для individual users, учебных программ и карьерных сервисов
